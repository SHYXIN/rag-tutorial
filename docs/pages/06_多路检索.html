多路检索

Multi-Query Retrieval — 1 个问题变 N 个检索词，提升召回率

🎯 问题背景

用户提问往往是模糊的、口语化的，和文档中的表述方式差异很大。

问题示例

用户问："那个公主后来咋样了？"
模糊、口语、指代不明
↓ 直接检索
文档写："沈令月于永和三年册封为嫡公主..."
语义差距大，可能检索不到

单路检索只用用户原始问题去搜索，容易因为表述差异而遗漏相关文档。

💡 核心思路

多路检索的核心思想：用 LLM 将 1 个问题改写成 3-5 个不同角度的检索词，分别检索后合并去重。

多路检索流程

用户原始问题："那个公主后来咋样了？"
↓ LLM 改写
Query 1: "沈令月的结局是什么？"
Query 2: "公主沈令月的后续经历"
Query 3: "沈令月最终命运如何？"
↓ 分别检索向量数据库
结果 1: [doc_A, doc_C, doc_E]
结果 2: [doc_B, doc_C, doc_F]
结果 3: [doc_A, doc_D, doc_G]
↓ 合并去重
最终结果: [doc_A, doc_B, doc_C, doc_D, doc_E, doc_F, doc_G]
7 个唯一文档，比单路检索多出 3 个

💻 实现代码

方案一：QueryRewriter + 手动合并

from langchain_core.output_parsers import StrOutputParser
from langchain_core.prompts import ChatPromptTemplate

# 1. 定义 Query 改写 Prompt
rewrite_prompt = ChatPromptTemplate.from_template("""
你是一个检索优化助手。请将用户的问题改写成 {n} 个不同角度的检索查询。
每个查询应该从不同角度表达相同的意图，以提高检索召回率。

用户问题：{question}

请输出 {n} 个查询，每行一个：
""")

# 2. 构建改写链
n_queries = 3
rewriter = rewrite_prompt | llm | StrOutputParser()

# 3. 执行多路检索
def multi_query_retrieve(question, retriever, n=3):
    # 改写查询
    rewritten = rewriter.invoke({"question": question, "n": n})
    queries = [q.strip() for q in rewritten.strip().split("\n") if q.strip()]
    queries.append(question)  # 保留原始问题

    # 分别检索 + 合并去重
    all_docs = []
    seen_ids = set()
    for q in queries:
        docs = retriever.invoke(q)
        for doc in docs:
            if doc.id not in seen_ids:
                seen_ids.add(doc.id)
                all_docs.append(doc)
    return all_docs

方案二：LangChain MultiQueryRetriever（推荐）

from langchain.retrievers.multi_query import MultiQueryRetriever

# 一行代码搞定
retriever = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(search_kwargs={"k": 5}),
    llm=llm,
    prompt=rewrite_prompt  # 可选：自定义改写 Prompt
)

# 使用方式与普通 retriever 完全一致
docs = retriever.invoke("那个公主后来咋样了？")

📊 效果对比

维度单路检索多路检索
检索次数1 次3-5 次
召回文档数3-5 个10-20 个（去重后）
召回率一般显著提升
延迟低（~200ms）较高（~600ms-1s）
LLM 调用0 次（仅生成答案）1 次（改写）+ 1 次（生成）
适用场景问题明确、表述规范问题模糊、口语化

🎯 适用场景

用户问题模糊 — 如"那个东西怎么用"、"后来怎么了"等口语化表达
需要高召回率 — 如法律检索、医疗问答，不能遗漏关键信息
文档表述多样 — 同一概念在不同文档中有不同说法
不适合的场景 — 问题已经很精确（如"第 3 章第 5 节的内容"）、对延迟敏感的场景
