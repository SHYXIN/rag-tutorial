混合检索（Hybrid Search）

BM25 关键词检索 + 向量语义检索，取长补短

🎯 问题背景：两种检索各有盲区

在 RAG 系统中，最常见的两种检索方式各有致命弱点：

两种检索的盲区对比

BM25 关键词检索
✅ 精确匹配关键词
❌ 不懂语义相似
🆚
向量语义检索
✅ 理解语义相似
❌ 不擅长精确匹配
场景BM25向量检索
用户搜"沈令月"，文档写"沈令月"✅ 精确匹配✅ 能匹配
用户搜"公主"，文档写"沈令月"❌ 关键词不同✅ 语义相近
用户搜"永和五年"，文档写"永和五年"✅ 精确匹配❌ 数字/日期向量区分度低
用户搜"谢初官职"，文档写"谢初任太傅"❌ 关键词不完全重合✅ 语义相近
用户搜"ISBN 978-7-xxx"✅ 精确匹配❌ ISBN 向量无意义
核心结论：BM25 擅长精确匹配（专有名词、数字、代码），向量检索擅长语义匹配（同义词、概念、描述）。两者互补，缺一不可。

💡 核心思路：BM25 + 向量取并集

混合检索的核心思想很简单：同时运行 BM25 和向量检索，然后合并结果。

混合检索流程

用户问题："沈令月在永和五年嫁给了谁？"
↓ 并行执行两种检索
BM25 检索
匹配关键词
"沈令月""永和五年"
向量检索
语义匹配
"公主""下嫁"
↓ 合并结果（去重 + 排序）
合并结果：取 BM25 Top-5 + 向量 Top-5 → 去重 → 重排序 → 返回 Top-5

📐 BM25 原理：TF-IDF 公式

BM25 基于经典的 TF-IDF 思想，但做了改进。其核心公式：

BM25(q, d) Σ TF(qi, d) × IDF(qi)

其中：
┌─────────────────────────────────────────────────────────┐
│ TF(qi, d) = (freq × (k1 + 1)) / (freq + k1 × (1 - b + b × |d|/avgdl)) │
│                                                         │
│ IDF(qi)   = log((N - n(qi) + 0.5) / (n(qi) + 0.5) + 1) │
└─────────────────────────────────────────────────────────┘

参数说明：
• freq    : 词 qi 在文档 d 中出现的次数
• |d|     : 文档 d 的长度（词数）
• avgdl   : 语料库中文档的平均长度
• N       : 语料库中文档总数
• n(qi)   : 包含词 qi 的文档数
• k1      : 词频饱和度参数（通常 1.2~2.0），控制词频的贡献上限
• b       : 文档长度归一化参数（通常 0.75），控制长度的影响

BM25 直觉理解

词频（TF）：一个词在文档中出现次数越多，越相关
但有上限——出现 100 次不等于出现 10 次的 10 倍重要
+
逆文档频率（IDF）：一个词在所有文档中越常见，越不重要
"的"出现在所有文档中，IDF ≈ 0；"永和五年"只出现在少数文档中，IDF 很高
+
长度归一化：长文档天然有更多词，需要惩罚
避免 10000 字文档仅仅因为词多就排第一

💻 实现代码

from langchain.retrievers import BM25Retriever, EnsembleRetriever
from langchain_community.vectorstores import Chroma

# 1. 准备文档
documents = [
    "沈令月是当朝嫡公主，性格温婉，深受皇帝宠爱。",
    "谢初在长林盛宴中拔得头筹，被皇帝封为太傅。",
    "永和五年，沈令月下嫁谢初，举国同庆。",
    "长林盛宴是三年一度的文武比试，胜者可获得皇帝赏赐。",
    "谢初出身寒门，凭借才华一步步晋升至太傅之位。",
]

# 2. 构建 BM25 检索器
bm25_retriever = BM25Retriever.from_documents(documents)
bm25_retriever.k = 5  # 返回 Top-5

# 3. 构建向量检索器
vectorstore = Chroma.from_documents(documents, embedding_function)
vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

# 4. 构建混合检索器（核心！）
hybrid_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.5, 0.5]  # BM25 和向量的权重，可调
)

# 5. 检索
results = hybrid_retriever.invoke("沈令月在永和五年嫁给了谁？")
for i, doc in enumerate(results):
    print(f"[{i+1}] {doc.page_content}")
权重调优：weights=[0.5, 0.5] 是起点。如果查询多为关键词精确匹配（如代码、产品名），可调高 BM25 权重；如果查询多为自然语言问题，可调高向量权重。
EnsembleRetriever 的合并策略：默认使用 RRF（Reciprocal Rank Fusion），即根据每个检索器返回的排名计算综合分数，而非简单拼接。

🎯 适用场景

通用知识库 — 用户查询类型多样，既有精确关键词也有自然语言，混合检索是最稳妥的选择
企业文档检索 — 文档中既有专业术语（BM25 擅长）也有描述性内容（向量擅长）
代码检索 — 函数名、变量名需要精确匹配，代码注释和描述需要语义匹配
电商搜索 — 商品型号、品牌名需要精确匹配，用户描述需要语义匹配

为什么混合检索后还要 Rerank？：混合检索的合并策略（如 RRF）是启发式的，不够精确。Rerank 使用 Cross-Encoder 对每个文档-问题对进行精细打分，精度更高。
成本权衡：混合检索增加了一次 BM25 查询（几乎零成本），Rerank 增加了 Cross-Encoder 推理成本。在精度要求高的场景下，这个投入是值得的。
