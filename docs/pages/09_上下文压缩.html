上下文压缩（Context Compression）

从检索到的文档中提取关键信息，去除噪音

🎯 问题背景：检索到的文档有噪音

向量检索返回的是文档块（chunk），而不是精确的答案。一个 500 字符的 chunk 中，可能只有 1-2 句话与问题相关，其余都是噪音。

检索结果中的噪音问题

用户问题："沈令月嫁给了谁？"
↓ 检索返回的文档块（500 字符）
"沈令月是当朝嫡公主，性格温婉，深受皇帝宠爱。她自幼聪慧，精通琴棋书画。
永和五年，沈令月下嫁谢初，举国同庆。
婚礼当天，十里红妆，百姓夹道围观。谢初身穿红色喜服，骑着高头大马..."
↓ 问题
❌ 只有 1 句话与问题相关，其余 80% 是噪音
❌ 噪音会干扰 LLM 的注意力，降低回答质量
❌ 浪费 Token 预算（更多文档 = 更高成本）
核心矛盾：检索粒度（chunk）和生成粒度（句子）不匹配。检索需要足够大的 chunk 以保证语义完整，但生成只需要最相关的句子。

💡 核心思路：提取关键句

上下文压缩的核心思想：不直接把检索到的文档块喂给 LLM，而是先提取与问题最相关的句子，只把精华部分提供给模型。

上下文压缩流程

检索返回 3 个文档块（共 ~1500 字符）
↓ Step 1: 压缩器处理
提取与问题相关的关键句（~400 字符）
↓ Step 2: 将压缩后的上下文提供给 LLM
LLM 基于精简上下文生成高质量回答

🔧 两阶段压缩策略

实际生产中，通常采用两阶段压缩：先用轻量级方法粗筛，再用 LLM 精筛。

两阶段压缩

检索返回 N 个文档块
↓ 阶段一：分数过滤（粗筛）
根据相似度分数过滤
保留分数 > 0.7 的块，丢弃低分块
成本低：纯数值比较，无需 LLM
↓ 阶段二：LLM 精选（精筛）
LLM 逐块阅读，提取与问题相关的句子
去除无关信息，保留核心内容
成本高：需要 LLM 调用，但精度更高
↓ 输出
压缩后的精炼上下文
阶段方法成本精度作用
阶段一（粗筛）相似度分数过滤几乎为零中等快速淘汰明显不相关的文档
阶段二（精筛）LLM 提取关键句1 次 LLM 调用高精确提取与问题相关的句子

💻 实现代码

from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor
from langchain_community.vectorstores import Chroma

# 1. 基础检索器
base_retriever = vectorstore.as_retriever(search_kwargs={"k": 10})

# 2. 构建压缩器（使用 LLM 提取关键句）
compressor = LLMChainExtractor.from_llm(llm)

# 3. 构建压缩检索器
compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=base_retriever
)

# 4. 使用
question = "沈令月在永和五年嫁给了谁？"
compressed_docs = compression_retriever.invoke(question)

# 5. 查看压缩效果
for i, doc in enumerate(compressed_docs):
    print(f"[{i+1}] {doc.page_content}")

# 6. 构建完整 RAG 链
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_template("""
请根据以下上下文回答问题：

上下文：{context}

问题：{question}

回答：
""")

rag_chain = (
    {"context": compression_retriever, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

answer = rag_chain.invoke("沈令月在永和五年嫁给了谁？")
print(answer)

🎯 适用场景

长文档检索 — 检索到的 chunk 很长，只有少量内容与问题相关
多文档融合 — 检索到多个文档，需要提取各自的关键信息后合并
Token 预算紧张 — 需要在有限的上下文窗口内塞入更多有效信息
高精度要求 — 噪音会显著降低回答质量的场景（如医疗、法律）

⚖️ 权衡对比

维度不压缩压缩
上下文完整性✅ 完整，不丢失信息⚠️ 可能丢失边缘信息
Token 成本❌ 高✅ 低（节省 40-60%）
回答准确率⚠️ 噪音可能干扰✅ 聚焦关键信息
额外延迟✅ 无⚠️ +1 次 LLM 调用（~300ms）
额外成本✅ 无⚠️ +1 次 LLM 调用
实现复杂度✅ 简单⚠️ 需要配置压缩器
经验法则：如果检索到的文档平均相关度 > 70%，可以不压缩；如果 < 50%，强烈建议压缩。大多数实际场景中，压缩带来的准确率提升远超过额外的 LLM 调用成本。
