HyDE 检索

Hypothetical Document Embeddings — 用"假答案"提升检索精度

🎯 问题背景：语义鸿沟

用户问题和文档之间存在语义鸿沟（Semantic Gap）：用户用"问句"提问，而文档用"陈述句"写内容。这两种表述在向量空间中的距离可能很远。

语义鸿沟示意

用户问题（问句）："公主最后嫁给了谁？"
↓ 向量空间中距离远
文档内容（陈述）："沈令月于永和五年下嫁谢初，举国同庆。"

HyDE 的核心洞察：用"答案"检索"答案"，比用"问题"检索"答案"更有效。

💡 核心思路

HyDE（Hypothetical Document Embeddings）的流程：先让 LLM 生成一个"假答案"（假设性文档），然后用这个假答案去检索真实文档。

HyDE 检索流程

用户问题："公主最后嫁给了谁？"
↓ Step 1: LLM 生成假答案
假答案："根据小说情节，沈令月公主最终嫁给了谢初。谢初在长林盛宴中表现出色，获得皇帝赏识，后被选为驸马。"
↓ Step 2: 对假答案做 Embedding
假答案向量：[0.12, -0.34, 0.56, ...]
↓ Step 3: 用假答案向量检索
匹配到真实文档："沈令月于永和五年下嫁谢初..."
假答案和真实文档都是陈述句，语义更接近！

🤔 为什么有效？

向量空间中的距离对比

❌ 问句 → 陈述句
"公主嫁给了谁？" → "沈令月下嫁谢初"
距离远，语义类型不同
对比
✅ 陈述句 → 陈述句
"公主嫁给了谢初..." → "沈令月下嫁谢初"
距离近，语义类型相同
关键原理：Embedding 模型在训练时，问题和答案的分布是不同的。HyDE 通过生成假答案，把"问题空间"映射到"答案空间"，消除了语义鸿沟。
假答案不需要正确：即使 LLM 生成的假答案是错的，只要它的"表述风格"和真实文档相似，就能提升检索效果。

💻 实现代码

from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# 1. 定义 HyDE Prompt
hyde_prompt = ChatPromptTemplate.from_template("""
请根据用户的问题，生成一段假设性的答案。
这段答案应该像真实文档一样，用陈述句详细描述相关信息。
不需要保证答案正确，但格式和风格应该像真实文档。

用户问题：{question}

假设性答案：
""")

# 2. 构建 HyDE 链
hyde_chain = hyde_prompt | llm | StrOutputParser()

# 3. HyDE 检索函数
def hyde_retrieve(question, retriever):
    # 生成假答案
    hypothetical_answer = hyde_chain.invoke({"question": question})
    print(f"假答案: {hypothetical_answer}")

    # 用假答案检索
    docs = retriever.invoke(hypothetical_answer)
    return docs

# 4. 使用
docs = hyde_retrieve("公主最后嫁给了谁？", vectorstore.as_retriever())

📊 效果对比

维度普通检索HyDE 检索
检索方式问题向量 → 检索文档问题 → 假答案 → 假答案向量 → 检索文档
语义匹配问句 vs 陈述句（鸿沟大）陈述句 vs 陈述句（鸿沟小）
检索精度一般更高（尤其对复杂问题）
额外延迟无+1 次 LLM 调用（~500ms）
额外成本无+1 次 LLM 调用
适用场景简单直接的问题复杂问题、语义鸿沟大的场景

🎯 适用场景

复杂问题 — 问题涉及多个概念、需要推理，直接检索效果差
语义鸿沟大 — 用户问题和文档表述风格差异大（如口语 vs 专业术语）
跨语言检索 — 用 LLM 生成目标语言的假答案，提升跨语言匹配
不适合的场景 — 简单事实性问题（如"作者是谁"）、对延迟极度敏感的场景

🔗 HyDE vs 多路检索

维度多路检索（Multi-Query）HyDE
核心思路1 个问题 → N 个检索词1 个问题 → 1 个假答案
解决什么问题问题模糊、表述单一语义鸿沟（问句 vs 陈述句）
检索次数N 次1 次
LLM 调用1 次（改写）1 次（生成假答案）
