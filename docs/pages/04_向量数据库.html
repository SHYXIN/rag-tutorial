向量数据库

存储和检索向量的专用数据库，RAG 的核心基础设施

🎯 什么是向量数据库？

传统数据库通过精确匹配查找数据（如 WHERE name = '沈令月'），而向量数据库通过相似度搜索查找数据。

传统数据库 vs 向量数据库

传统数据库
精确匹配
"沈令月" = "沈令月"
🆚
向量数据库
相似度搜索
"公主" ≈ "沈令月"

向量数据库的核心能力：给定一个查询向量，快速找到与其最相似的 K 个向量。这个操作称为 ANN（Approximate Nearest Neighbor，近似最近邻）搜索。

📐 核心概念

向量检索流程

查询文本
"沈令月的身份"
→
Embedding
文本 → 向量
→
ANN 搜索
在索引中查找
→
Top-K 结果
最相似的 K 个文档
概念说明
向量（Vector）由 Embedding 模型生成的高维数组，如 512 维、1024 维
索引（Index）加速检索的数据结构，如 HNSW、IVF、PQ
ANN 搜索近似最近邻搜索，牺牲少量精度换取大幅速度提升
Top-K返回相似度最高的 K 个结果
距离度量余弦相似度、欧氏距离、内积

🏗️ HNSW 索引原理（简化版）

HNSW（Hierarchical Navigable Small World）是目前最流行的向量索引算法，核心思想是分层图搜索。

HNSW 分层结构

第 2 层（顶层）：稀疏，只有少数节点，快速定位大致区域
↓ 逐层下降
第 1 层（中层）：中等密度，进一步缩小范围
↓ 逐层下降
第 0 层（底层）：包含所有节点，精确搜索最近邻
类比：就像查字典先看目录（顶层），再看章节（中层），最后翻到具体页（底层）。
关键参数：M（每层最大邻居数，影响精度和内存）、efConstruction（建索引时的搜索深度）、efSearch（查询时的搜索深度）。

💻 Chroma 使用示例

import chromadb
from chromadb.utils import embedding_functions

# 1. 创建客户端（本地持久化）
client = chromadb.PersistentClient(path="./chroma_db")

# 2. 创建集合（指定 Embedding 函数和距离度量）
ef = embedding_functions.SentenceTransformerEmbeddingFunction(
    model_name="BAAI/bge-small-zh-v1.5"
)
collection = client.get_or_create_collection(
    name="novel_chunks",
    embedding_function=ef,
    metadata={"hnsw:space": "cosine"}  # 余弦相似度
)

# 3. 添加文档
collection.add(
    documents=["沈令月是当朝嫡公主，性格温婉。", "谢初在长林盛宴中拔得头筹。"],
    metadatas=[{"chapter": 1, "source": "novel.txt"}, {"chapter": 2, "source": "novel.txt"}],
    ids=["chunk_001", "chunk_002"]
)

# 4. 检索
results = collection.query(
    query_texts=["公主是谁？"],
    n_results=3,
    where={"chapter": {"$lte": 5}}  # 可选：元数据过滤
)

# 5. 查看结果
for doc, score in zip(results['documents'][0], results['distances'][0]):
    print(f"[{score:.3f}] {doc}")

⚡ 性能调优参数

参数作用调大效果调小效果
M（HNSW）每层最大邻居数精度更高，内存更大内存更小，精度降低
efConstruction建索引搜索深度索引质量更高，建得更慢建索引更快，质量降低
efSearch查询时搜索深度召回率更高，查询更慢查询更快，召回率降低
nprobe（IVF）搜索的聚类数精度更高，速度更慢速度更快，精度降低
Top-K返回结果数上下文更丰富，可能含噪音上下文更精简，可能遗漏
经验值：M=16, efConstruction=200, efSearch=100 是一个不错的起点。
内存估算：100 万条 512 维向量 + HNSW 索引，大约需要 2-4 GB 内存。
