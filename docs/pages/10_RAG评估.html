RAG 评估（Evaluation）

系统化评估 RAG 系统的质量，找到优化方向

🎯 为什么需要评估？

RAG 系统是一个多组件 pipeline，任何一个环节出问题都会影响最终效果。没有评估，优化就是盲人摸象。

RAG 系统的故障链

切块策略不当 → 语义被切断
↓ 连锁反应
Embedding 质量差 → 检索结果不相关
↓ 连锁反应
检索结果不相关 → LLM 无法生成好答案
↓ 连锁反应
LLM 生成答案质量差 → 用户体验差
评估的核心价值：定位问题出在哪个环节——是检索没找到相关文档？还是 LLM 没有正确使用文档？还是文档本身质量差？
没有评估的后果：你优化了 Prompt，但问题其实出在切块策略上；你换了 Embedding 模型，但问题其实出在检索参数上。

📐 四个评估维度

RAG 评估需要从四个维度全面衡量，每个维度对应 pipeline 的不同环节：

四维评估框架

① 检索相关性（Retrieval Relevance）
检索到的文档是否与问题相关？——评估检索环节
↓ 检索结果作为上下文
② 答案忠实度（Answer Faithfulness）
答案是否基于检索到的文档，而非模型编造？——评估生成环节
↓ 答案与问题对比
③ 答案相关性（Answer Relevance）
答案是否回答了用户的问题？——评估整体效果
↓ 答案与标准答案对比
④ 答案完整性（Answer Completeness）
答案是否涵盖了所有必要信息？——评估覆盖度
维度评估什么评估方法对应环节
检索相关性检索到的文档是否相关人工打分 / LLM 打分检索层
答案忠实度答案是否有文档依据LLM 事实验证生成层
答案相关性答案是否回答了问题人工打分 / LLM 打分端到端
答案完整性答案是否覆盖所有要点与标准答案对比端到端

📏 评分标准

每个维度通常使用 0-1 或 1-5 的评分标准。以下是常用的 0-1 二分类标准：

维度1 分（通过）0 分（不通过）
检索相关性 检索到的文档中至少有一篇包含答案 所有检索到的文档都不包含答案
答案忠实度 答案中的每个事实都能在检索到的文档中找到依据 答案中存在无法被文档支持的事实（幻觉）
答案相关性 答案直接回答了用户的问题 答案偏离了问题，或答非所问
答案完整性 答案涵盖了标准答案中的所有关键信息 答案遗漏了重要信息
忠实度是最重要的维度：如果答案不忠实（有幻觉），即使相关性再高也是有害的。忠实度是 RAG 系统的底线。

💻 实现代码

from ragas import evaluate
from ragas.metrics import (
    faithfulness,        # 答案忠实度
    answer_relevancy,    # 答案相关性
    context_recall,      # 检索召回率
    context_precision,   # 检索精确度
)
from datasets import Dataset

# 1. 准备评估数据
# 每条数据包含：问题、答案、检索到的上下文、标准答案
eval_data = {
    "question": [
        "沈令月嫁给了谁？",
        "谢初的官职是什么？",
        "长林盛宴多久举办一次？",
    ],
    "answer": [
        "沈令月嫁给了谢初。",
        "谢初官至太傅。",
        "长林盛宴三年举办一次。",
    ],
    "contexts": [
        ["永和五年，沈令月下嫁谢初，举国同庆。"],
        ["谢初在长林盛宴中拔得头筹，被皇帝封为太傅。"],
        ["长林盛宴是三年一度的文武比试，胜者可获得皇帝赏赐。"],
    ],
    "ground_truth": [
        "沈令月在永和五年下嫁谢初。",
        "谢初被皇帝封为太傅。",
        "长林盛宴每三年举办一次。",
    ],
}

# 2. 转换为 Ragas 数据集
dataset = Dataset.from_dict(eval_data)

# 3. 执行评估
result = evaluate(
    dataset=dataset,
    metrics=[
        context_precision,   # 检索精确度：检索到的文档中有多少是相关的
        context_recall,      # 检索召回率：相关文档中有多少被检索到
        faithfulness,        # 答案忠实度：答案是否有文档依据
        answer_relevancy,    # 答案相关性：答案是否回答了问题
    ],
    llm=eval_llm,            # 用于评估的 LLM（可以用 GPT-4）
)

# 4. 查看结果
print(result)
# 输出示例：
# {
#     'context_precision': 0.917,
#     'context_recall':    0.833,
#     'faithfulness':      0.944,
#     'answer_relevancy':  0.912,
# }

# 5. 查看详细结果（每条数据的分数）
df = result.to_pandas()
print(df)

评估结果分析流程

查看综合分数
↓ 定位薄弱环节
检索分数低
→ 优化检索
换 Embedding / 调参数
忠实度低
→ 优化 Prompt
加强"基于文档回答"指令
相关性低
→ 优化检索 + Prompt
检查切块策略

🎯 适用场景

上线前评估 — 在正式发布前，用评估数据集验证系统是否达到预期质量
迭代优化 — 每次修改检索策略、Prompt、Embedding 模型后，对比评估分数变化
A/B 测试 — 对比两种方案（如 BM25 vs 混合检索）的量化效果差异
回归测试 — 更新知识库后，确保系统质量没有下降

⚠️ 注意事项

评估数据集质量：评估数据集的代表性直接决定评估结果的可靠性。建议至少准备 50-100 条覆盖各种问题类型的数据。
LLM 评估的偏差：用 LLM 评估 LLM 存在"自评偏差"。建议关键场景下用人工评估交叉验证。
不要只看平均分：平均分高不代表系统好。关注最差 case（P10 分数），这些才是用户实际遇到的糟糕体验。
评估成本：Ragas 的评估需要调用 LLM，100 条数据大约需要 200-400 次 LLM 调用。建议用较便宜的模型做日常评估，用 GPT-4 做关键节点评估。
维度之间的权衡：提高检索召回率（检索更多文档）可能降低精确度（引入更多噪音）。需要根据业务场景找到最佳平衡点。
