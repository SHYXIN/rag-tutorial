对话记忆（Conversation Memory）

让 RAG 系统拥有"记忆"，支持多轮对话

🎯 问题背景：三轮挑战

真实场景中，用户不会一次性把问题说清楚，而是通过多轮对话逐步深入。这给 RAG 系统带来三大挑战：

多轮对话的三大挑战

① 指代消解（Coreference Resolution）
用户说"他后来怎样了？"——"他"指谁？没有上下文就无法理解
② 上下文窗口限制（Context Window）
10 轮对话可能有 5000+ tokens，超出模型的上下文窗口
③ 信息压缩（Information Compression）
不能简单地把所有历史对话都塞进 Prompt，需要智能压缩
轮次用户输入问题无记忆系统的回答
第 1 轮"沈令月是谁？"清晰"沈令月是当朝嫡公主。"
第 2 轮"她嫁给了谁？""她"指代不明"请问您说的'她'是指谁？"
第 3 轮"他们什么时候结婚的？""他们"指代不明"请问您说的'他们'是指谁？"
第 4 轮"他的官职是什么？""他"指代不明"请问您说的'他'是指谁？"
核心需求：RAG 系统需要"记住"对话历史，才能正确理解后续问题中的指代和省略。

🏗️ 三层记忆架构

借鉴人类记忆的三层模型（感觉记忆、短期记忆、长期记忆），对话记忆也采用三层架构：

三层记忆架构

短期记忆：滑动窗口（Sliding Window）
保留最近 N 轮对话原文，不做任何处理
作用：保留最新的对话细节，用于理解指代
容量：最近 3-5 轮（~1000-2000 tokens）
↓ 超出窗口的历史
中期记忆：摘要压缩（Summary Compression）
将超出滑动窗口的历史对话压缩为摘要
作用：保留对话的主要话题和结论
容量：1-3 句话的摘要（~100-200 tokens）
↓ 跨会话的信息
长期记忆：实体追踪（Entity Tracking）
追踪对话中提到的关键实体及其属性
作用：跨会话记住用户关注的人物、事件
容量：实体列表（~50-100 tokens）
记忆层存储内容容量更新频率实现方式
短期（滑动窗口）最近 N 轮原文~1500 tokens每轮追加双端队列
中期（摘要压缩）历史对话摘要~150 tokens每 N 轮更新LLM 摘要
长期（实体追踪）关键实体列表~80 tokens每轮更新NER + 属性提取

💻 实现代码

from collections import deque
from langchain_core.messages import HumanMessage, AIMessage

class ConversationMemory:
    """三层对话记忆系统"""

    def __init__(self, llm, window_size=3):
        self.llm = llm
        self.window_size = window_size
        # 短期记忆：滑动窗口，保留最近 N 轮原文
        self.short_term = deque(maxlen=window_size * 2)
        # 中期记忆：摘要
        self.mid_term_summary = ""
        # 长期记忆：实体追踪
        self.long_term_entities = {}

    def add_message(self, human_msg: str, ai_msg: str):
        """添加一轮对话"""
        self.short_term.append(HumanMessage(content=human_msg))
        self.short_term.append(AIMessage(content=ai_msg))
        self._update_entities(human_msg, ai_msg)
        if len(self.short_term) == self.short_term.maxlen:
            self._compress_to_mid_term()

    def _update_entities(self, human_msg, ai_msg):
        """提取并更新实体"""
        known_entities = {
            "沈令月": {"type": "人物", "身份": "嫡公主", "配偶": "谢初"},
            "谢初": {"type": "人物", "官职": "太傅", "配偶": "沈令月"},
        }
        for name, attrs in known_entities.items():
            if name in human_msg or name in ai_msg:
                self.long_term_entities[name] = attrs

    def _compress_to_mid_term(self):
        """将短期记忆压缩为中期摘要"""
        history_text = "\n".join(
            f"{'用户' if isinstance(m, HumanMessage) else 'AI'}: {m.content}"
            for m in self.short_term
        )
        prompt = f"请用 1-2 句话总结以下对话的核心内容：\n{history_text}"
        self.mid_term_summary = self.llm.invoke(prompt).content
        self.short_term.clear()

    def get_context(self) -> str:
        """获取完整的记忆上下文"""
        parts = []
        if self.long_term_entities:
            desc = "\n".join(f"- {n}：{a}" for n, a in self.long_term_entities.items())
            parts.append(f"已知实体：\n{desc}")
        if self.mid_term_summary:
            parts.append(f"对话摘要：{self.mid_term_summary}")
        if self.short_term:
            recent = "\n".join(
                f"{'用户' if isinstance(m, HumanMessage) else 'AI'}: {m.content}"
                for m in self.short_term
            )
            parts.append(f"最近对话：\n{recent}")
        return "\n\n".join(parts)

    def rewrite_question(self, question: str) -> str:
        """基于记忆重写问题（指代消解）"""
        context = self.get_context()
        if not context:
            return question
        prompt = f"""根据以下对话历史，将用户的新问题改写为自包含问题。

对话历史：
{context}

用户新问题：{question}

改写后的问题："""
        return self.llm.invoke(prompt).content


# 使用
memory = ConversationMemory(llm, window_size=3)
memory.add_message("沈令月是谁？", "沈令月是当朝嫡公主。")
rewritten = memory.rewrite_question("她嫁给了谁？")
print(f"改写后：{rewritten}")
# 输出："沈令月嫁给了谁？"（自动补全了"她"的指代）

以下展示 5 轮对话中，三层记忆的内容变化：

轮次用户问题改写后的问题短期记忆中期记忆长期记忆

第 4 轮时的记忆状态

中期记忆："用户询问了沈令月的身份，得知她是当朝嫡公主。"

🎯 适用场景

客服机器人 — 用户会多次追问同一个订单/问题，需要记住上下文
知识问答助手 — 用户围绕一个主题深入提问，后续问题依赖前面的回答
代码助手 — 用户逐步描述需求，后续修改依赖前面的代码上下文
不适合的场景 — 每次对话完全独立（如一次性问答），加记忆反而增加成本和延迟

⚙️ 生产优化建议

优化方向策略效果
降低延迟摘要压缩和实体提取与 LLM 生成并行执行节省 ~300ms
降低 Token实体列表用结构化格式（JSON）而非自然语言节省 ~40% Token
提高指代消解准确率用轻量级 NER 模型替代 LLM 提取实体更快更稳定
跨会话记忆将实体列表持久化到数据库，下次对话加载支持长期用户画像
记忆清理定期清理不活跃的实体，避免记忆膨胀控制 Token 增长
滑动窗口大小：window_size=3 是大多数场景的起点。指代密集的场景（如代码讨论）可调大到 5，简单问答可减小到 2。
摘要质量：中期记忆的摘要质量直接影响指代消解效果。建议在摘要 Prompt 中明确要求"保留所有人物和关键事实"。
