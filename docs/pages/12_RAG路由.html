RAG 路由（Router）

根据问题类型自动选择最优检索策略

🎯 问题背景：不同问题适合不同策略

在实际应用中，用户的问题千差万别——有的需要精确查找，有的需要语义理解，有的甚至不需要检索。如果所有问题都走同一套检索流程，要么精度不够，要么浪费资源。

同一套策略处理不同问题的困境

精确查询："谢初的官职是什么？"
用向量检索 → 可能把"太傅"和"太傅府"混淆，不够精确
语义查询："公主最后嫁给了谁？"
用关键词检索 → "公主""嫁给"可能匹配不到"沈令月下嫁谢初"
简单查询："你好"
用任何检索 → 完全没必要，直接让 LLM 回复即可
核心思路：在检索之前，先对问题进行分类，然后根据类型路由到不同的处理策略——这就是 RAG 路由。

💡 核心思路：问题分类 → 选择策略

RAG 路由的本质是一个决策层，它在用户问题和检索策略之间充当"交通指挥官"：

路由决策流程

用户问题输入
↓
问题分类器
判断问题类型
↓ 根据类型路由
精确查询
→ BM25 检索
语义查询
→ 向量检索
简单查询
→ 直接 LLM

📋 三种问题类型详解

类型特征示例问题最佳策略
精确查询 包含专有名词、数字、日期、代码等需要精确匹配的内容 "谢初的官职？"
"永和五年发生了什么？"

📐 路由规则表

路由器的核心是一张规则表，定义了从问题特征到检索策略的映射：

判断条件路由目标原因
问题长度 < 5 字 且无实体直接 LLM太短，大概率是闲聊或问候
包含数字/日期/专有名词BM25 检索精确匹配比语义匹配更可靠
包含"什么是"/"解释"/"介绍"直接 LLM 或向量检索通用知识问题，可能不需要检索
包含"怎么"/"如何"/"为什么"向量检索需要理解语义和因果关系
包含代词（他/她/它/他们）先指代消解 → 向量检索需要上下文理解
包含"多少"/"几个"/"第几"BM25 + 向量混合既需要精确匹配也需要语义理解
规则优先级：精确匹配规则优先于语义匹配规则。如果问题同时满足多个条件，优先选择 BM25 检索。
兜底策略：如果路由不确定，默认走混合检索（BM25 + 向量），确保覆盖率。

💻 实现代码

from enum import Enum
from langchain.retrievers import BM25Retriever
from langchain_community.vectorstores import Chroma

class QueryType(Enum):
    EXACT = "exact"       # 精确查询
    SEMANTIC = "semantic" # 语义查询
    SIMPLE = "simple"     # 简单查询

class QuestionRouter:
    """问题路由器：根据问题类型选择检索策略"""

    def __init__(self, bm25_retriever: BM25Retriever,
                 vector_retriever, llm=None):
        self.bm25 = bm25_retriever
        self.vector = vector_retriever
        self.llm = llm

    def classify(self, question: str) -> QueryType:
        """规则-based 问题分类"""
        # 简单查询：太短且无意义
        if len(question.strip()) < 5:
            return QueryType.SIMPLE

        # 精确查询：包含数字、日期、专有名词
        import re
        if re.search(r'\d{4}年|\d+月|\d+日|ISBN|编号|代码', question):
            return QueryType.EXACT

        # 精确查询：包含引号（用户明确指定了关键词）
        if '"' in question or "'" in question:
            return QueryType.EXACT

        # 语义查询：包含疑问词
        if re.search(r'怎么|如何|为什么|什么|谁|哪|多少|几个', question):
            return QueryType.SEMANTIC

        # 默认：语义查询
        return QueryType.SEMANTIC

    def route(self, question: str):
        """根据分类结果路由到对应策略"""
        query_type = self.classify(question)

        if query_type == QueryType.SIMPLE:
            return {"type": "simple", "results": []}

        elif query_type == QueryType.EXACT:
            docs = self.bm25.invoke(question)
            return {"type": "exact", "results": docs}

        else:  # SEMANTIC
            docs = self.vector.invoke(question)
            return {"type": "semantic", "results": docs}


class RouterRAG:
    """带路由的 RAG 系统"""

    def __init__(self, router: QuestionRouter, llm):
        self.router = router
        self.llm = llm

    def ask(self, question: str) -> str:
        # 1. 路由决策
        route_result = self.router.route(question)
        query_type = route_result["type"]
        docs = route_result["results"]

        # 2. 根据类型生成回答
        if query_type == "simple":
            prompt = f"用户说：{question}\n请直接回复："
        else:
            context = "\n".join(d.page_content for d in docs[:5])
            prompt = f"""根据以下文档回答问题。

文档：
{context}

问题：{question}
回答："""

        return self.llm.invoke(prompt).content


# 使用示例
router = QuestionRouter(bm25_retriever, vector_retriever, llm)
rag = RouterRAG(router, llm)

# 不同类型的问题自动路由
print(rag.ask("你好"))              # → 直接 LLM
print(rag.ask("谢初的官职？"))      # → BM25 检索
print(rag.ask("公主嫁给了谁？"))    # → 向量检索

🎯 适用场景

多知识库场景 — 用户问题可能涉及不同领域的知识库（产品文档、技术文档、FAQ），路由可以自动选择正确的知识库
混合查询场景 — 用户既有精确查找（订单号、产品编号）也有自然语言问题，路由能自动区分
成本敏感场景 — 通过路由跳过不必要的检索，降低 API 调用成本和延迟
不适合的场景 — 问题类型单一的系统（如纯 FAQ 问答），路由反而增加复杂度

🚀 进阶：用 LLM 做路由

规则-based 路由简单高效，但覆盖面有限。更强大的方式是用 LLM 本身来做路由决策：

class LLMRouter:
    """使用 LLM 进行智能路由"""

    def __init__(self, llm, retrievers: dict):
        """
        retrievers: {
            "product_db": product_retriever,
            "tech_docs": tech_retriever,
            "faq": faq_retriever,
        }
        """
        self.llm = llm
        self.retrievers = retrievers

    def route(self, question: str) -> str:
        """让 LLM 决定使用哪个检索器"""
        prompt = f"""你是一个路由助手。根据用户问题，选择最合适的知识库。

可选知识库：
- product_db: 产品信息、价格、规格
- tech_docs: 技术文档、API 参考、代码示例
- faq: 常见问题、操作指南、故障排除
- none: 不需要检索（闲聊、通用知识）

用户问题：{question}

请只输出知识库名称（product_db/tech_docs/faq/none）："""

        target = self.llm.invoke(prompt).content.strip()
        return target

    def ask(self, question: str) -> str:
        target = self.route(question)

        if target == "none" or target not in self.retrievers:
            return self.llm.invoke(question).content

        docs = self.retrievers[target].invoke(question)
        context = "\n".join(d.page_content for d in docs[:5])
        prompt = f"根据以下文档回答问题。\n\n文档：{context}\n\n问题：{question}\n回答："
        return self.llm.invoke(prompt).content
LLM 路由的优势：能理解复杂语义，处理规则难以覆盖的边界情况（如"帮我查一下上次那个订单"）
LLM 路由的代价：每次问题都要额外调用一次 LLM，增加延迟和成本。可以用小模型（如 GPT-3.5）来降低成本。
混合方案：先用规则快速判断（简单查询直接返回），规则无法判断时再调用 LLM 路由，兼顾效率和准确性。
