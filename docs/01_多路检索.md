# 多路检索（Multi-Query Retrieval）

## 问题背景

用户的问题往往很模糊：
- "那个技术怎么用？" → 哪个技术？怎么用是什么意思？
- "她为什么选他？" → 她是谁？他是谁？

单路检索只用一个 query 去匹配，容易漏掉相关文档。

## 核心思路

**把 1 个问题改写成 3-5 个不同角度的检索词，分别检索，合并结果。**

```
原始问题 → LLM 改写 → [query1, query2, query3]
                              ↓        ↓        ↓
                          检索1    检索2    检索3
                              ↓        ↓        ↓
                           合并去重 → 最终文档列表
```

## 实现

```python
class QueryRewriter:
    SYSTEM_PROMPT = """生成 3 个检索 query，从不同角度覆盖问题。每行一个，不要解释。"""

    def rewrite(self, query: str) -> list[str]:
        response = call_llm([
            {"role": "system", "content": self.SYSTEM_PROMPT},
            {"role": "user", "content": f"问题：{query}"}
        ])
        return [q.strip() for q in response.strip().split('\n') if q.strip()]


class MultiQueryRetriever:
    def retrieve(self, query: str) -> list:
        # 改写
        queries = self.rewriter.rewrite(query)
        # 分别检索 + 合并去重
        seen, docs = set(), []
        for q in queries:
            for doc in self.db.similarity_search(q, k=3):
                key = doc.page_content[:100]
                if key not in seen:
                    seen.add(key)
                    docs.append(doc)
        return docs
```

## 效果对比

```
问题："沈令月为什么要选谢初当驸马？"

单路检索：3 个文档（可能漏掉）
多路检索：5-7 个文档（覆盖面更广）

改写的 query 示例：
  1. 沈令月选谢初当驸马的原因
  2. 沈令月选择谢初作为驸马的动机
  3. 谢初被选为驸马的影响
```

## 适用场景

- 用户问题模糊、有多种理解方式
- 知识库较大，单 query 容易漏掉相关文档
- 对召回率要求高（宁可多召回，不能漏掉）

## 代价

- 每次检索要多调 1 次 LLM（改写 query）
- 检索次数 ×3（每个 query 都要检索）
- 适合对精度要求高、对延迟不敏感的场景
