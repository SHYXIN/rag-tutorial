# 文档切块（Document Chunking）

## 为什么需要切块？

LLM 的 context window 有限，不能把整篇文档塞进去。

```
一篇小说 = 400,000 字符
LLM context window = 4,096 token ≈ 6,000 字符

→ 必须切成小块，每次只检索相关的小块
```

## 切块策略

### 1. 固定长度切块（最简单）

```python
splitter = RecursiveCharacterTextSplitter(
    chunk_size=300,        # 每块 300 字符
    chunk_overlap=30,      # 块之间重叠 30 字符
    separators=["\n\n", "\n", "。", "！", "？", " ", ""]
)
```

- `chunk_size`：每块最大字符数
- `chunk_overlap`：相邻块之间的重叠字符数（防止语义被切断）
- `separators`：按优先级尝试切分（先按段落，再按句子，最后按字符）

### 2. 语义切块（更智能）

按语义边界切分，而不是固定长度：
- 按段落切分（`\n\n`）
- 按句子切分（`。！？`）
- 按主题切分（需要 LLM 辅助）

### 3. 递归切块（推荐）

LangChain 的 `RecursiveCharacterTextSplitter`：
1. 先尝试按 `\n\n`（段落）切分
2. 如果段落太长，按 `\n`（换行）切分
3. 如果还太长，按 `。`（句子）切分
4. 最后按字符切分

## 切块大小的影响

| 块大小 | 优点 | 缺点 |
|--------|------|------|
| 小（100-200） | 检索精准，噪音少 | 可能丢失上下文，需要更多块 |
| 中（300-500） | 平衡精准度和上下文 | 通用选择 |
| 大（800-1000） | 上下文完整 | 可能包含无关内容，检索不够精准 |

## 重叠的重要性

```
文档："...沈令月选谢初为驸马。谢初在长林盛宴中拔得头筹..."

无重叠：
  块1: "...沈令月选谢初为驸马。"
  块2: "谢初在长林盛宴中..."
  → 检索"谢初"时，块1 可能因为不包含"谢初"的更多描述而漏掉

有重叠（overlap=30）：
  块1: "...沈令月选谢初为驸马。谢初在..."
  块2: "谢初在长林盛宴中拔得头筹..."
  → 两个块都包含"谢初"，不会漏掉
```

## 实际经验

- **小说/故事**：按段落切分，chunk_size=300-500，overlap=30-50
- **技术文档**：按章节/段落切分，chunk_size=500-800
- **FAQ**：每个问答作为一块（不需要再切）
- **代码**：按函数/类切分

## 常见问题

**Q: 块太小怎么办？**
A: 检索时多召回几个块（增大 k），或者用更大的 chunk_size

**Q: 块太大怎么办？**
A: 用上下文压缩，从块中提取关键句

**Q: 怎么知道块大小合不合适？**
A: 看检索结果——如果检索到的块里大量内容无关，说明块太大；如果答案不完整，说明块太小
