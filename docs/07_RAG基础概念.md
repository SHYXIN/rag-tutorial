# RAG 基础概念

## 什么是 RAG？

**RAG（Retrieval-Augmented Generation，检索增强生成）** = 检索系统 + 生成模型

核心思想：**在让 LLM 回答问题之前，先从外部知识库中检索相关信息，然后把检索结果和问题一起提供给模型。**

```
用户问题
    ↓
[检索] 从知识库中找到相关文档
    ↓
[增强] 把文档 + 问题拼成 prompt
    ↓
[生成] LLM 基于文档生成回答
    ↓
返回答案
```

## 为什么需要 RAG？

LLM 的局限性：
1. **知识截止** — 训练数据有截止日期，不知道最新信息
2. **幻觉** — 可能编造不存在的事实
3. **私有知识** — 不知道你的企业内部文档
4. **不可追溯** — 答案来源不明

RAG 的优势：
1. **知识可更新** — 更新知识库即可，不需要重新训练模型
2. **答案有据可查** — 可以追溯到具体文档
3. **处理私有知识** — 可以检索企业内部文档
4. **不需要微调** — 直接用现有模型

## RAG 完整流程

### 离线阶段（知识库构建）

```
原始文档（PDF/Word/网页/Markdown...）
    ↓
[文档解析] 提取纯文本
    ↓
[文档切块] 切成适合检索的小块（200-500 字符）
    ↓
[Embedding] 文本 → 向量（1536 维）
    ↓
[向量数据库] 存储向量 + 原文 + 元数据
    ↓
知识库就绪 ✅
```

### 在线阶段（用户提问）

```
用户问题
    ↓
[Embedding] 问题 → 向量
    ↓
[检索] 相似度搜索，找到 Top K 相关文档
    ↓
[上下文压缩] 提取关键句，减少噪音
    ↓
[构建 Prompt] 系统提示 + 文档 + 问题
    ↓
[LLM 生成] 基于文档生成回答
    ↓
[引用溯源] 标注答案来源
    ↓
返回答案 ✅
```

## 核心组件

| 组件 | 作用 | 常用选择 |
|------|------|----------|
| 文档解析 | 提取纯文本 | PyPDFLoader, TextLoader |
| 文档切块 | 切分成小块 | RecursiveCharacterTextSplitter |
| Embedding | 文本转向量 | BGE, OpenAI Embeddings |
| 向量数据库 | 存储和检索向量 | Chroma, Milvus, Pinecone |
| LLM | 生成回答 | GPT-4, Claude, LongCat |

## RAG vs 微调

| | RAG | 微调 |
|--|-----|------|
| 知识更新 | 更新知识库即可 | 需要重新训练 |
| 答案可追溯 | ✅ 可追溯到文档 | ❌ 无法追溯 |
| 成本 | 低（不需要训练） | 高（需要训练资源） |
| 适用场景 | 知识库问答、文档检索 | 特定任务优化、风格调整 |

**经验法则：**
- 知识需要频繁更新 → 用 RAG
- 需要改变模型行为/风格 → 用微调
- 两者可以结合：先用微调优化模型，再用 RAG 提供知识
